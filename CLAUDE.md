# aws-lambda-runtime

A lightweight Clojure library (`com.thoughtsforge/aws-lambda-runtime`) that implements the AWS Lambda Runtime API polling loop. Published to Clojars. Also ships a `deps-new` project template in the same repo.

## Project Structure

```
src/thoughtsforge/
  aws_lambda_runtime.clj              — public API: start!, post-init-error!
  aws_lambda_runtime/impl/
    http.clj                          — JDK HttpClient wrapper (GET/POST), no extra deps
    context.clj                       — Lambda invocation headers → context map

test/thoughtsforge/
  aws_lambda_runtime_test.clj         — runtime tests (with-redefs on http layer)
  aws_lambda_runtime/impl/
    context_test.clj                  — pure unit tests
    http_test.clj                     — real JDK HttpServer mock

resources/io/github/estebanleonsoto/aws_lambda_runtime/
  template.edn                        — deps-new template descriptor
  root/                               — files that go to generated project root
  src/{{main/file}}.clj               — handler template
  test/{{main/file}}_test.clj         — test template

docs/
  getting-started.md                  — full scaffold → deploy walkthrough

build.clj                             — tools.build: jar, install, deploy
```

## Key Design Decisions

- **Handler injection via HOF** — `(runtime/start! handler-fn)`. No dynamic namespace resolution, no `Class/forName`, no env var parsing. The developer passes their handler function directly from `-main`. This is intentional: Ring-style, GraalVM-friendly (no reflection for handler lookup), errors surface at compile time.
- **No extra HTTP dep** — uses JDK `java.net.http.HttpClient` exclusively. Keeps the library GraalVM-compatible and dependency-free beyond Clojure + cheshire.
- **`environ` was dropped** — the original project used it for dev handler config. Removed in favor of plain `System/getenv`. Dev handler config is now the developer's responsibility.
- **impl namespaces are internal** — `thoughtsforge.aws-lambda-runtime.impl.*` is not part of the public API. Only `thoughtsforge.aws-lambda-runtime` (`start!` and `post-init-error!`) is public.

## Public API

```clojure
(require '[thoughtsforge.aws-lambda-runtime :as runtime])

(runtime/start! handler-fn)        ;; starts polling loop, blocks indefinitely
(runtime/post-init-error! throwable) ;; reports fatal init error before System/exit
```

## Build & REPL

```bash
clj -M:dev                         # start nREPL on port 7888
clj -M:test -m kaocha.runner       # run tests
clj -T:build jar                   # build jar
clj -T:build install               # install to ~/.m2 for local testing
clj -T:build deploy                # deploy to Clojars (needs CLOJARS_USERNAME + CLOJARS_PASSWORD)
```

## nREPL

Port is fixed at **7888** via `.nrepl.edn`. Start with `clj -M:dev`. nREPL discovery: `clj-nrepl-eval --discover-ports`.

## Testing Approach

- `context_test` — pure unit tests, no infrastructure
- `http_test` — real HTTP round-trip using JDK `com.sun.net.httpserver.HttpServer` (no extra test deps)
- `runtime_test` — `with-redefs` on `thoughtsforge.aws-lambda-runtime.impl.http/get!` and `http/post!` to control the Lambda Runtime API responses without a real server. The loop is stopped by having `http/get!` throw on the second call (after one successful invocation).

## Versioning

Schema: `MAJOR.MINOR(2-digit).PATCH(3-digit)` — e.g., `0.01.000`, `0.01.001`, `1.00.000`.
Current version: `0.01.000` (defined in `build.clj`).

## deps-new Template

The template lives in `resources/io/github/estebanleonsoto/aws_lambda_runtime/` (included in `:paths` via `"resources"`). Users scaffold with:

```bash
clj -Tnew create \
  :template io.github.estebanleonsoto/aws-lambda-runtime \
  :name myorg/my-lambda
```

Template substitution variables used: `{{top/ns}}`, `{{top/file}}`, `{{main/ns}}`, `{{main/file}}`, `{{group/id}}`, `{{artifact/id}}`.
